📄 PRODUCT REQUIREMENTS DOCUMENT (PRD)
Product: Playto Payout Engine (Lean MVP)

Design and implement a minimal payout engine that allows merchants to:

View balances derived from a ledger
Request payouts
Track payout lifecycle asynchronously

The system must guarantee:

Financial correctness
Concurrency safety
Idempotent request handling
Reliable async processing (without external infra)

Freelancers/agencies need a system to:

Track incoming funds
Withdraw earnings
Avoid duplicate or inconsistent payouts

7. ⚙️ Functional Requirements
7.1 Ledger System
Store all transactions as CREDIT or DEBIT
Balance = SUM(CREDIT) - SUM(DEBIT)
Must be computed using database queries only
7.2 Payout Request API
Endpoint:
POST /api/v1/payouts
Headers:
Idempotency-Key: <UUID>
Body:
{
  "amount_paise": 5000,
  "bank_account_id": "bank_123"
}
Behavior:
Check idempotency key
Start DB transaction
Lock merchant ledger rows
Validate balance
Insert DEBIT (hold funds)
Create payout (status = pending)
Store response in idempotency table
Trigger async job
7.3 Idempotency Handling
Key is unique per merchant
If key exists → return stored response
Prevent duplicate payouts
Handle concurrent duplicate requests safely
7.4 Concurrency Handling
Scenario:

Two payout requests at same time

Expected:
Only one succeeds
Implementation:
Use select_for_update() inside transaction
Prevent race condition
7.5 Payout Processing (Async Worker)
Tool:
Django-Q (ORM-based queue)
Flow:
Fetch pending payout
Update → processing
Simulate:
70% success
20% failure
10% stuck
Outcomes:
Success:
status → completed
Failure:
status → failed
Insert CREDIT (refund)
7.6 Retry Mechanism
If payout in processing > 30 seconds:
Retry
Increase attempts
Backoff:
2s → 4s → 8s
Max attempts: 3
After max → mark failed + refund
7.7 State Machine
Allowed:
pending → processing → completed
pending → processing → failed
Disallowed:
completed → any state
failed → completed
Enforcement:
Validate before every update
Reject invalid transitions
